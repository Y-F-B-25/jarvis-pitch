The Problem

AI today is a brilliant amnesiac

Most people use AI the same way: open a chat, ask a question, get an answer, close the window. Tomorrow you start over. The AI doesn't remember what you told it yesterday. It doesn't know what your other conversations were about. It can't talk to itself. You end up being the one holding all the context, doing all the coordination, repeating yourself constantly.

You're getting help from the smartest intern in the world — but an intern who shows up every morning with no memory of yesterday, no idea what their coworkers are working on, and no way to hand off their notes to the next shift.

What if AI could be a team instead of a chatbot?

Under the Hood

How the system actually works

Plain English first. Click any "Technical details" section if you want to see exactly how it's built.

1
Every agent starts by reading the master index

There's a single document that tells every agent where everything is: which projects exist, who's working on what, where to find past work, and what rules to follow. When an agent starts a new session, it reads this index first. It never asks me for context it can find itself.

Technical details →
The master index is BRAIN_INDEX.md, a structured Markdown file in an Obsidian vault. It contains project locations, agent session registry, file paths to every key document, and 13 codified governance rules. The vault itself has 100+ files organized into 10 numbered sections (Home, Security, Architecture, Projects, Operations, Sessions, Decisions, Contacts, Handoffs, VaultBus). Agents query specific files on demand rather than loading the entire vault.
2
Agents coordinate through a structured message system

Benji sends instructions to other agents through a messaging system. Each agent has an inbox. Messages are tracked from creation through delivery to completion. If something gets stuck, the system flags it. There's a complete audit trail of every message ever sent.

Technical details →
The coordination protocol is called VaultBus. It has five layers: Status (each agent's current state), Commands (directives from Benji to agents), Escalations (blockers needing attention), Events (append-only audit log in NDJSON), and Views (human-readable dashboards). A Python daemon backed by SQLite provides durable message routing with a strict lifecycle state machine: created → queued → delivered → acknowledged → completed/failed/expired. Messages are idempotent (SHA-256 checksum tracking prevents duplicates). Commands are immutable after creation.
3
Context survives forever through structured handoffs

AI conversations have a limit on how much they can hold in memory. When an agent approaches that limit, it writes a detailed handoff document — what it accomplished, what's still pending, what decisions were made, and exact next steps. The next version of that agent starts by reading the handoff and picks up exactly where the last one left off. The Website agent is on version 12. It inherited everything from versions 1 through 11 without loading a single old conversation.

Technical details →
This is the Context Ceiling Protocol. Agents track their tool call count from session start. At 60 calls they enter "yellow zone" and notify the user. At 80 calls they hard-stop: checkpoint current task, update their VaultBus status file, write a structured handoff document (standardized template covering accomplishments, pending work, context, next steps, lessons learned), and update the session README. The handoff is then incorporated into the next version's boot prompt. Version numbers reflect usage intensity, not calendar time — a heavily used agent versions up quickly, a light-use agent stays at v1. There are currently 16 handoff documents in the vault, forming a continuous knowledge chain.
4
Agents can take real actions on my computer

AI agents typically run in a sandbox — they can read and write files but can't actually do things like push code to GitHub or run a build process. I have a secure bridge that lets agents send specific, pre-approved commands to my actual machine. The Website agent uses this to commit code, push it to GitHub, and trigger deployments on Vercel — all without me touching anything.

Technical details →
The bridge is called the Watcher — a Python script (cowork_watcher_v2.py) running in a terminal on the host machine. It polls a JSON inbox file for commands, executes them locally (where git credentials, dev tools, and network access are available), and writes results to an outbox file. Security: requires an auth token on every command, enforces a command allowlist, blocks dangerous patterns (rm -rf, sudo, curl | bash), rate-limits to 2-second intervals, and logs everything. All git remote operations (push, pull, fetch) go through the watcher — never attempted from the sandbox.
5
Quality is enforced through 13 governance rules

Every rule in the system exists because something went wrong without it. An early version of the Website agent ran too long, forgot its own protocols, and tried to push code through a method that couldn't work. Now there's a hard limit on session length. Another time, an agent created a useful file but didn't register it anywhere — the next agent couldn't find it. Now every file creation is paired with immediate registration. These aren't theoretical best practices. They're scar tissue turned into policy.

Technical details →
The 13 rules are codified in BRAIN_INDEX.md and enforced through boot prompts, handoff docs, and Benji's monitoring. Key examples: Rule 6 — Create + Register is atomic (any new file must be registered in BRAIN_INDEX in the same action). Rule 10 — Context Ceiling (hard stop at 80 tool calls, non-negotiable). Rule 12 — Use Edit for existing files, not full rewrites (prevents invisible bugs). Rule 5 — Zero mechanical work for the human. There's also a Vault Lint system (inspired by Andrej Karpathy's LLM wiki patterns) that periodically scans the knowledge base for structural issues: orphan files, broken references, stale information, and contradictions between documents.
Governance

Six Key Rules

Part of a 13-rule framework learned the hard way. Each rule exists because something went wrong without it.

01

Context Ceiling

Every agent tracks tool call count. Yellow at 60, hard stop at 80. Agents write handoffs before they degrade — never after. No context is ever lost.

02

Create + Register = Atomic

Any new file must be registered in BRAIN_INDEX in the same action. If it's not indexed, it doesn't exist to the next agent.

03

Zero Mechanical Work

Yacob makes decisions. Everything operational — file management, deployments, coordination, monitoring — is the agents' problem.

04

Versioned Handoffs

When an agent hits its ceiling, it writes a structured handoff. The next version boots with full context. Knowledge compounds, never resets.

05

Vault Lint

Automated quality checks (Karpathy-inspired) catch orphaned files, broken references, and contradictions. The knowledge base stays clean as it grows.

06

Watcher-Gated Execution

No direct git push from sandbox. All external operations go through the watcher bridge — logged, authenticated, rate-limited, and auditable.

Security & Trust

Hardened by Design

The system is built with real security constraints, not afterthoughts.

Authenticated Watcher

Every command to the Mac requires a token. Blocked command patterns (rm -rf, sudo, eval, curl|bash). 34-command allowlist. 2-second rate limit between executions.

Sandboxed Agents

Each agent runs in an isolated Linux container. Can read/write shared files but cannot access the network, other sessions, or system resources directly.

Immutable Commands

VaultBus command files are immutable after creation (except status field). Full audit trail in SQLite event log. No agent can edit another agent's status file.

Credential Isolation

API keys, tokens, and sensitive data stored in a dedicated vault folder (01 — Security/). Never embedded in boot prompts or shared across agents.

Why This Matters

This isn't what you've seen before

Every week there's a new "AI agent" demo. Most of them book flights, send emails, and break after three steps. Project Jarvis is something fundamentally different.

Not Automation

Creative reasoning, not task execution

These agents don't fill out forms. They make design decisions informed by reference sites. They direct a film through a multi-tool AI pipeline. They synthesize legal strategy from five different attorneys. This is judgment work, not button-pressing.

Not Ephemeral

Weeks of accumulated context

Most AI conversations last one session. Project Jarvis agents inherit context from every previous version. The Website agent at v13 carries institutional knowledge from v1. Nothing is ever lost. The system gets smarter over time.

Not Fragile

Governance that prevents degradation

AI agents degrade when they run too long. We discovered this, built the Context Ceiling Protocol, and now it's enforced as law. Every failure we've encountered became a rule. The system hardens itself.

Not Narrow

Five domains, one system

Web development, filmmaking, legal case management, AI tool evaluation, knowledge infrastructure. The coordination patterns are domain-agnostic. If you can describe the work, you can build an agent for it.

Not Theoretical

Running in production, daily

This isn't a proof of concept or a weekend project. It's a live system that has run 50+ sessions, produced 16 structured handoffs, shipped 12 website versions, and managed a real legal case with real deadlines.

Not Dependent

Built on standard tools, no proprietary lock-in

Obsidian, Notion, Python, SQLite, GitHub, Vercel. No custom cloud infrastructure. No expensive APIs. The entire system runs on a Mac desktop with a mounted folder. Portable, reproducible, and affordable.
